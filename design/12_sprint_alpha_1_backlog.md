# 12. Sprint Alpha-1 Backlog

## 목표

`Sprint Alpha-1`의 목적은 현재 프로토타입 루프를 "처음 시작한 플레이어가 길을 잃지 않고 첫 복원 목표를 이해하는 상태"까지 끌어올리는 것이다.

이번 스프린트는 새 시스템을 많이 늘리는 것보다 아래 3가지를 우선한다.

- 첫 세션의 진입 장벽 제거
- 허브 복원 시스템의 데이터 기반 골격 확정
- 허브 HUD를 실제 플레이 흐름 중심으로 재정렬

## Sprint Goal

- 신규 저장을 시작하면 첫 원정 진입, 첫 복귀, 첫 복원/업그레이드까지 자연스럽게 이어진다.
- 허브 구역 상태를 코드 상수가 아니라 데이터 자산과 저장 데이터로 관리한다.
- 허브 화면에서 플레이어가 "지금 해야 할 다음 행동"을 즉시 이해할 수 있다.

## 범위

### 포함

1. `A1` 첫 플레이 튜토리얼 플로우
2. `B1` 허브 복원 구역 데이터 구조
3. `E1` 허브 HUD 정리

총합: `15 Story Points`

### 제외

- 실제 허브 복원 소비 액션 구현 (`B2`)
- 지구별 목표 타입 확장 (`C1`, `C2`)
- 메인 메뉴/세이브 슬롯 (`E3`)
- 엔딩, 자유 복원, 접근성 옵션

## 추천 기간

- 솔로 기준: `1.5~2주`
- 2인 기준: `4~6일`

## 작업 순서

1. `B1` 허브 복원 구역 데이터 구조
2. `A1` 첫 플레이 튜토리얼 플로우
3. `E1` 허브 HUD 정리

이 순서를 권장하는 이유는 튜토리얼과 HUD가 모두 "현재 허브 상태를 무엇으로 읽는가"에 의존하기 때문이다.

## 백로그 상세

### Backlog Item 1. 허브 복원 구역 데이터 구조

- ID: `B1`
- 추정: `5점`
- 목적:
  - 허브 복원 진행을 업그레이드와 분리된 독립 시스템으로 확장할 준비를 끝낸다.
- 구현 범위:
  - `HubZoneDef` ScriptableObject 정의 추가
  - 허브 구역 목록을 `Resources/ScriptableObjects`에서 읽도록 경로 정의
  - 저장 데이터에 구역별 상태 값 추가
  - 허브 UI/매니저가 구역 상태를 읽는 접근 계층 추가
- Acceptance Criteria:
  - [ ] 구역 정의가 코드 상수 대신 데이터 자산으로 존재한다
  - [ ] 저장 데이터가 구역별 상태를 유지한다
  - [ ] 허브 진입 시 구역 상태를 읽어올 수 있다
  - [ ] 아직 시각 변화가 없더라도 텍스트/상태값으로 확인 가능하다
- 완료 산출물:
  - 신규 데이터 타입
  - 기본 구역 자산 세트
  - 저장/로드 연동

### Backlog Item 2. 첫 플레이 튜토리얼 플로우

- ID: `A1`
- 추정: `5점`
- 목적:
  - 첫 세션에서 플레이어가 허브, 원정, 복귀, 첫 강화/복원 루프를 설명 없이 통과하게 만든다.
- 구현 범위:
  - 신규 저장 여부 판정
  - 튜토리얼 단계 enum 또는 상태값 정의
  - 허브 첫 진입 시 목표 문구/버튼 유도
  - 첫 원정 복귀 후 다음 행동 유도
  - 튜토리얼 완료 후 일반 루프로 전환
- Acceptance Criteria:
  - [ ] 신규 저장에서 튜토리얼이 자동 시작된다
  - [ ] 튜토리얼 중 현재 해야 할 행동이 HUD 또는 패널에 표시된다
  - [ ] 첫 원정 복귀 이후 허브에서 다음 행동 안내가 바뀐다
  - [ ] 완료 이후 동일 저장에서는 튜토리얼이 반복되지 않는다
- 완료 산출물:
  - 튜토리얼 상태 저장
  - 허브/원정 결과를 잇는 안내 문구
  - 첫 세션 완료 조건

### Backlog Item 3. 허브 HUD 정리

- ID: `E1`
- 추정: `5점`
- 목적:
  - 현재 디버그 성격이 강한 허브 화면을 "다음 행동 안내 + 핵심 자원 + 핵심 버튼" 구조로 단순화한다.
- 구현 범위:
  - 허브 HUD에서 우선순위 낮은 버튼 제거 또는 디버그 전용으로 격리
  - 핵심 CTA를 `원정 시작`, `복원/업그레이드 확인`, `현재 목표` 중심으로 재배치
  - 튜토리얼 단계 또는 현재 진행 목표를 HUD 상단에 표시
  - 구역 상태 요약을 허브 UI에 노출
- Acceptance Criteria:
  - [ ] 허브 진입 시 핵심 행동 2~3개만 강조된다
  - [ ] 현재 목표 또는 추천 행동이 명시적으로 보인다
  - [ ] 자원/지구 진행/복원 상태가 한 화면에서 확인 가능하다
  - [ ] 디버그 성격 조작은 기본 사용자 동선에서 빠진다
- 완료 산출물:
  - 허브 HUD 레이아웃 개편
  - 진행 목표 문구 영역
  - 구역 상태 요약 영역

## Definition Of Done

- 코드 기준:
  - 저장 데이터 스키마가 새 필드를 안정적으로 처리한다
  - 허브 UI가 새 데이터 구조를 읽는다
  - 튜토리얼 상태 전환이 단일 흐름으로 정리된다
- 검증 기준:
  - 신규 저장으로 `허브 -> 첫 원정 -> 결과 -> 허브`를 한 번 통과했을 때 진행이 막히지 않는다
  - 기존 저장을 불러와도 치명적 오류가 나지 않는다
  - Unity 콘솔 에러 `0`
  - EditMode 테스트 회귀 없음

## QA 체크리스트

1. 신규 저장 시작 시 튜토리얼 상태가 활성화된다.
2. 첫 허브 화면에서 해야 할 행동이 즉시 보인다.
3. 첫 원정 진입 버튼 또는 CTA가 하나의 주 동선으로 정리돼 있다.
4. 첫 복귀 후 허브 목표 문구가 다음 단계로 바뀐다.
5. 저장 후 재실행해도 튜토리얼 상태가 유지된다.
6. 허브 구역 상태가 저장/로드 이후에도 유지된다.
7. 기존 저장을 불러왔을 때 기본값 또는 마이그레이션 처리로 깨지지 않는다.

## 리스크

### 리스크 1. 저장 데이터 변경 회귀

대응:
- 신규 필드는 기본값 경로를 가진다
- 기존 저장 로드 테스트를 반드시 포함한다

### 리스크 2. 런타임 생성형 UI의 복잡도 증가

대응:
- 완전한 비주얼 리디자인이 아니라 정보 우선순위 재배치에 집중한다
- 튜토리얼 강조는 색/문구/정렬 수준에서 해결한다

### 리스크 3. 허브 복원 시스템의 책임 혼재

대응:
- 이번 스프린트는 데이터 구조까지만 확정한다
- 실제 소비/연출은 다음 스프린트 `B2`에서 처리한다

## 스프린트 종료 조건

- `B1`, `A1`, `E1`의 Acceptance Criteria를 모두 충족한다.
- 다음 스프린트에서 바로 `B2`와 `C1/C2`를 착수할 수 있을 정도로 데이터 구조와 UI 골격이 안정화된다.
- 문서상 목표가 아니라 실제 프로젝트 상태로 "첫 30분 경험"이 한 단계 전진했다고 판단할 수 있다.
